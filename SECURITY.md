# سياسة أمان صحتنا

## المسار المعتمد للقاعدة

نفّذ فقط، وبالترتيب:

1. `supabase-schema.sql`
2. `supabase-production-hardening.sql`
3. `supabase-rls-tests.sql`

ملف التحصين موحّد وقابل لإعادة التشغيل. الملفات الأمنية القديمة متوقفة عمداً حتى لا تعيد سياسات ضعيفة أو تتلف البيانات المشفرة.

## نموذج الوصول

| العملية | المسار المسموح |
|---|---|
| قراءة العيادات والأطباء والمواعيد المتاحة | عروض عامة محدودة لا تكشف بيانات المرضى |
| إنشاء حجز | `create_appointment` فقط |
| استرجاع حجز المريض | UUID الحجز + رقم الهاتف عبر `get_patient_booking` |
| إلغاء حجز المريض | UUID الحجز + رقم الهاتف عبر `cancel_patient_booking` |
| تقييم طبيب | موعد مكتمل ومطابق عبر `create_verified_review` |
| عمل العيادة | مستخدم Supabase Auth مرتبط بعيادة، عبر RPCs مقيّدة |
| عمل الإدارة | مستخدم Supabase Auth موجود في `admin_users` |

لا توجد صلاحيات مباشرة عامة لإدراج أو تعديل الحجوزات أو الجداول أو التقييمات. سياسات RLS تبقى طبقة حماية إضافية، والدوال الحساسة تستخدم `SECURITY DEFINER` مع `search_path` ثابت وفحوص صريحة للهوية والملكية.

## حماية البيانات

- تُشفّر ملاحظات المريض وبيانات الهوية بـ `pgcrypto`.
- مفتاح التشفير يُنشأ مرة واحدة داخل Supabase Vault، ولا توجد دالة عامة تعيده للعميل.
- ترقية البيانات تتأكد من نجاح فك التشفير قبل مسح النص الأصلي، وتتوقف إذا وجدت بيانات مشفرة غير قابلة للتحقق.
- العرض العام للتقييمات يخفي هوية المريض ولا يعرض رقم الموعد.
- رموز التفعيل لا تظهر في العرض العام، وتُلغى بعد الاستخدام.
- يمنع تحديد المعدل التخمين المتكرر لرموز التفعيل ومعرّفات الحجوزات وإنشاء الحجوزات بكثافة.

## المصادقة والإعداد

- لا توجد حسابات إنتاج بكلمات مرور ثابتة؛ التحصين يحذف حسابات العرض القديمة إن وُجدت.
- أنشئ مستخدمي Auth الحقيقيين من لوحة Supabase أو مسار دعوة موثوق.
- اربط أول مدير فقط عبر `private.bootstrap_admin` من SQL Editor.
- استخدم `anon key` العام في `js/runtime-config.js`، ولا تستخدم `service_role key` في المتصفح مطلقاً.
- غيّر مفاتيح أو رموز ظهرت سابقاً في سجل Git قبل الإطلاق.

## اختبارات RLS

`supabase-rls-tests.sql` يعمل كأمر ذري واحد، وينظف بياناته عند النجاح أو يتراجع عنها تلقائياً عند الفشل، ويختبر:

- عدم كشف بيانات المرضى للزائر.
- عزل كل عيادة عن الأخرى.
- منع التعديلات المباشرة من العميل.
- التحقق من ملكية الحجز عند الاسترجاع والإلغاء والتقييم.
- منع الحجز المزدوج والقيم غير الصالحة.
- صلاحيات الإدارة وسجل التدقيق.

شغّل الملف بعد كل تعديل للسكيما. أي نتيجة `FAIL` تمنع النشر.

## قائمة ما قبل الإنتاج

- [ ] جميع اختبارات RLS تعرض `PASS` على نسخة مرحلية.
- [ ] الحساب الإداري الحقيقي مرتبط، والحسابات التجريبية غير موجودة.
- [ ] متغيرات التشغيل تحتوي فقط على URL و`anon key` الصحيحين.
- [ ] فُعّل تأكيد البريد أو سياسة دعوات مناسبة لحسابات العيادات.
- [ ] أُعيدت مفاتيح Supabase التي سبق كشفها أو مشاركتها.
- [ ] النسخ الاحتياطي والاسترجاع جُرّبا مع مفتاح Vault.
- [ ] سجلات التدقيق والتنبيهات ومعدلات الأخطاء تحت المراقبة.
- [ ] تم اختبار الحجز والإلغاء والتفعيل وتغيير الحالة من الواجهات على بيئة مرحلية.

## الإبلاغ عن ثغرة

لا تنشر تفاصيل أو بيانات مرضى داخل Issue عامة. أرسل وصفاً مختصراً وخطوات إعادة الإنتاج دون أسرار، ثم شارك التفاصيل الحساسة عبر قناة خاصة يحددها مالك المستودع.
