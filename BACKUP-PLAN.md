# 🗄️ صحتنا - خطة النسخ الاحتياطي وإدارة المفاتيح

## نظرة عامة

هذه الخطة تضمن استرداد البيانات والمفاتيح في حالات الفشل أو الكوارث.

## 📅 جدول النسخ الاحتياطي

| النوع | التكرار | الاحتفاظ | الموقع |
|------|---------|----------|--------|
| **قاعدة البيانات الكاملة** | يومياً (2:00 AM) | 30 يوماً | Supabase Dashboard → Database → Backups |
| **النسخ الأسبوعية** | كل أحد | 12 أسبوعاً | Supabase Storage (مضغوط) |
| **النسخ الشهرية** | أول كل شهر | 12 شهراً | تخزين خارجي (S3/Google Drive) |
| **مفاتيح التشفير** | عند التدوير | دائم (لا تُحذف) | Vault + نسخة خارجية |

## 🔑 إدارة مفاتيح التشفير

### المفتاح الحالي
- **الموقع:** Supabase Vault (`vault.secrets` → `sahatna_field_key`)
- **النوع:** 256-bit hex (64 محرف)
- **الوصول:** فقط عبر `get_encryption_key()` SECURITY DEFINER

### النسخة الاحتياطية للمفتاح
⚠️ **مهم جداً:** يجب حفظ المفتاح في مكان آمن **خارج** قاعدة البيانات.

```
المواقع المقترحة:
1. مدير كلمات المرور (1Password / Bitwarden) — مشاركة مع فريق الإدارة
2. خزنة مادية (Safe) — نسخة ورقية في ظرف مختوم
3. لا تُخزّن في git أو أي مستودع كود أبداً
```

### تدوير المفتاح (Key Rotation)

**الجدول:** كل 90 يوماً

**الخطوات:**
1. احفظ المفتاح الحالي كنسخة احتياطية (في حال فشل التدوير)
2. شغّل في Supabase SQL Editor:
   ```sql
   SELECT rotate_encryption_key();
   ```
3. ستقوم الدالة بـ:
   - توليد مفتاح جديد عشوائي
   - إعادة تشفير كل البيانات بالمفتاح الجديد
   - تحديث السر في Vault
4. احفظ المفتاح الجديد في مدير كلمات المرور
5. اختبر فك التشفير:
   ```sql
   SELECT decrypt_field(patient_notes_encrypted) FROM appointments LIMIT 1;
   ```

### في حال فقدان المفتاح
🚨 **كارثة:** لا يمكن استرداد البيانات المشفّرة بدون المفتاح

**الإجراء:**
1. استرجع المفتاح من النسخة الاحتياطية (مدير كلمات المرور / الخزنة)
2. إذا فُقد نهائياً:
   - البيانات المشفّرة (`patient_notes_encrypted`, `national_id_encrypted`) **غير قابلة للاسترداد**
   - باقي البيانات (أسماء، هواتف، مواعيد) سليمة في النسخ الاحتياطية
3. أنشئ مفتاحاً جديداً وأعد تشفير البيانات الجديدة فقط

## 🔄 استرداد البيانات (Disaster Recovery)

### سيناريو 1: فشل قاعدة البيانات
1. اذهب إلى Supabase Dashboard → Database → Backups
2. اختر آخر نسخة احتياطية ناجحة
3. اضغط "Restore"
4. تأكد من تشغيل جميع ملفات SQL migration بالترتيب

### سيناريو 2: فشل Supabase بالكامل
1. أنشئ مشروع Supabase جديد
2. شغّل ملفات SQL بالترتيب:
   ```
   1. supabase-schema.sql
   2. supabase-security-hardening.sql
   3. supabase-field-encryption.sql
   4. supabase-vault-migration.sql
   5. fix-booking-rls.sql
   6. supabase-migration-functions.sql
   ```
3. استورد البيانات من آخر نسخة احتياطية شهرية
4. حدّث `js/supabase-config.js` بالـ URL والمفتاح الجديد

### سيناريو 3: اختراق أمني
1. **أوقف الوصول فوراً:** أعد تعيين كلمات مرور Supabase
2. **افحص سجل التدقيق:** `SELECT * FROM audit_log ORDER BY created_at DESC LIMIT 100;`
3. **دوّر مفاتيح التشفير:** `SELECT rotate_encryption_key();`
4. **دوّر مفاتيح API:** Supabase Dashboard → Settings → API → Reset keys
5. **حدّث `js/supabase-config.js`** بالمفاتيح الجديدة
6. **افحص RLS:** شغّل `supabase-rls-tests.sql`
7. **أبلغ المستخدمين المتأثرين**

| البيانات | مُشفّر؟ | قابل للاسترداد؟ |
|----------|---------|-----------------|
| أسماء المرضى | ❌ | ✅ نعم |
| أرقام الهواتف | ❌ | ✅ نعم |
| المواعيد والحجوزات | ❌ | ✅ نعم |
| ملاحظات طبية | ✅ (Vault key) | ✅ مع المفتاح / ❌ بدونه |

| كلمات مرور المستخدمين | ✅ (Supabase Auth) | ✅ نعم (bcrypt) |
